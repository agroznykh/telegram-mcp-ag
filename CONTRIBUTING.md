# Contributing

## Setup

```bash
python3 -m venv .venv
.venv/bin/python -m pip install -e ".[dev]"
```

## Tests

```bash
.venv/bin/python -m pytest
```

Tests don't touch the network and don't need a real Telegram session.

## Building the Claude Desktop bundle

```bash
bash mcpb/build.sh
```

Requires Node.js (the `mcpb` CLI runs via `npx`, nothing to install ahead of
time) and, ideally, `uv` (used to refresh `mcpb/uv.lock` before packing ŌĆö the
build still works without it as long as an `uv.lock` already exists). Output:
`dist/telegram-mcp-ag-<version>.mcpb`.

`mcpb/src/telegram_mcp_ag/` is generated by this script (copied from
`src/telegram_mcp_ag/`) and is git-ignored ŌĆö don't edit it directly, edit
`src/telegram_mcp_ag/` and rebuild.

The script refuses to build if:
- `pyproject.toml`'s `version` and `mcpb/manifest.json`'s `version` disagree
  (users see the manifest version but get the package version ŌĆö a mismatch
  would go unnoticed);
- `mcpb/pyproject.toml`'s `dependencies` don't cover everything listed in the
  top-level `pyproject.toml` (a missing dependency in the bundle means an
  `ImportError` the first time someone opens Claude Desktop, not at build time).

## Updating the pinned upstream (`chigwell/telegram-mcp`)

This project imports `telegram_mcp.runtime` and `telegram_mcp.runner`
directly ŌĆö that module is the *only* integration point with upstream, and it
carries no compatibility guarantee across commits. Treat every bump as a
manual review, not a routine dependency update:

1. Pick the new commit SHA from
   [`chigwell/telegram-mcp`](https://github.com/chigwell/telegram-mcp).
2. Update the pin in **both** places ŌĆö they must always match:
   - `pyproject.toml` ŌåÆ `dependencies` ŌåÆ the `telegram-mcp @ git+...@<sha>` line
   - `mcpb/pyproject.toml` ŌåÆ same line
3. Diff what changed in `telegram_mcp/runtime.py` and `telegram_mcp/tools/`
   between the old and new SHA, in particular:
   - the signature of anything `src/telegram_mcp_ag/server.py` imports by name
     (`ensure_connected`, `get_client`, `get_sender_name`, `resolve_entity`,
     `sanitize_user_content`, `validate_id`, `with_account`, `ToolAnnotations`,
     `json_serializer`, `log_and_format_error`, `mcp`);
   - `_apply_exposed_tools_mode` / the `readOnlyHint` filtering that backs
     `TELEGRAM_EXPOSED_TOOLS=read-only` ŌĆö this is the read-only guarantee the
     whole project is built on;
   - the `TranscribeAudioRequest` response shape `server.py` reads
     (`trial_remains_num`, `trial_remains_until_date`, `pending`, `text`) and
     the `transcribe_audio_trial_weekly_number` / `transcribe_audio_trial_duration_max`
     keys read from `help.getAppConfig`.
4. Run `pytest`, then rebuild and smoke-test the `.mcpb` bundle
   (`bash mcpb/build.sh`, install it locally, connect a real account).
5. Update the SHA mentioned in `CLAUDE.md` and `PLAN.md` if either references it.

## Installer scripts

`install.sh` (macOS/Linux) and `install.ps1` (Windows) target people who have
never opened a terminal, so their output stays in Russian even though code
comments are in English ŌĆö keep that split when editing them. Sanity-check
syntax before committing:

```bash
bash -n install.sh
shellcheck install.sh   # if installed
```

```powershell
Invoke-ScriptAnalyzer -Path install.ps1   # if PSScriptAnalyzer is installed
```

Both scripts are idempotent by design (`install.sh` again shouldn't create a
second Claude Code registration) ŌĆö preserve that when adding new registration
steps.
